# cloud-storage / frontend

React 18 + Vite + TypeScript SPA. Talks to `api-gateway` over HTTP for
auth, file storage, share links, and the AI assistant.

## Stack

- **React 18** + **TypeScript 5** strict
- **Vite 5** (dev server + production bundler)
- **TanStack Router** (file-based routing, autogenerated route tree)
- **TanStack Query** (server-state cache, mutations, refresh)
- **Zustand** (client-only state: auth, uploads, AI modal)
- **Tailwind CSS v3** + custom design tokens (Dropbox-style dark theme)
- **shadcn/ui** primitives (Dialog, Select, Dropdown, etc.) on top of Radix
- **react-hook-form** + **zod** for form validation
- **sonner** for toasts
- **lucide-react** for icons

## Layout

```
src/
  api/             — typed API clients (auth, storage, share, ai)
  components/      — reusable UI (layout shell, ui/ primitives, common)
  features/        — feature folders (auth, files, files/upload, share, ai)
  store/           — Zustand stores
  lib/             — pure helpers (cn, env, errors, format, mime, id)
  routes/          — TanStack Router file-based routes
  routeTree.gen.ts — autogenerated; do NOT edit by hand
deployments/
  nginx.conf       — runtime web server config (CSP, SPA fallback, caching)
  docker-entrypoint.sh — writes /config.js from env at container start
  k8s/             — Deployment, Service, Ingress, ConfigMap
Dockerfile         — multi-stage Node → Nginx
```

## Local development

Requires Node.js 20+ and npm 10+.

```bash
npm install
cp .env.example .env.local      # set VITE_API_BASE_URL to your gateway
npm run dev                     # http://localhost:5173
```

Backend services (`auth-service`, `storage-service`, `ai-service`,
`api-gateway`) must be reachable at `VITE_API_BASE_URL`. See each
service's README for setup; for an end-to-end smoke test you also need
PostgreSQL and MinIO running.

## Routes

| path                  | purpose                                  |
| --------------------- | ---------------------------------------- |
| `/login`              | auth: sign in                            |
| `/register`           | auth: create account                     |
| `/files`              | redirects to `/files/{user-root-id}`     |
| `/files/$folderId`    | files page for a given folder            |
| `/share/$token`       | public read-only / read-write share view |

`/files/*` is auth-gated by `<AuthGate>`; missing tokens redirect to
`/login?next=<encoded-path>`. Public-share is intentionally token-only;
no Bearer is sent (`apiPublicFetch`).

## API mapping

See `src/api/*.ts`. Every endpoint listed in the api-gateway routes file
(`api-gateway/internal/presentation/http/router.go`) is wrapped with a
typed function. `apiFetch` (auth) and `apiPublicFetch` (no auth) both:

1. Unwrap the `ServerEnvelope` `{ status, data | error, code }` shape.
2. On `401` for `apiFetch`, mutex-protected refresh via
   `/auth/v1/refresh`, then retry once.
3. Surface domain codes (`NODE_NOT_FOUND`, `MOVE_INTO_DESCENDANT`,
   `LLM_UNAVAILABLE`, …) as `ApiError.code`; `lib/errors.ts` maps each to
   a user-facing Russian message for toasts.

## Auth & token storage

Per `frontend-blueprint.md` §6.1, MVP uses `localStorage`:

- `cs:access_token`, `cs:refresh_token`, `cs:user`
- The auth store hydrates on page load and listens to `storage` events
  for cross-tab sync.
- Migrating to `httpOnly` cookies later means rewriting `lib/tokens.ts`
  and a small change in `api/client.ts`; nothing else.

## Upload flow

Three phases — see `features/files/upload/uploadFile.ts`:

1. `POST /storage/v1/files/upload-url` → presigned PUT URL + headers.
2. **XHR** `PUT` to S3 (we use XHR not fetch because fetch lacks
   upload-progress events). Progress is throttled to
   `VITE_UPLOAD_PROGRESS_THROTTLE_MS`.
3. `POST /storage/v1/files/{id}/finalize` with `size_bytes` and
   `checksum` (SHA-256 hex of the whole file, computed via Web Crypto).

A worker pool (`uploadQueue.ts`) drains queued entries up to
`VITE_UPLOAD_PARALLELISM`. Cancellation aborts the XHR and best-effort
calls `POST /storage/v1/files/{id}/abort` so pending blobs don't ghost.

## Build & deploy

```bash
docker build \
  --build-arg VITE_API_BASE_URL=https://gw.example.com \
  -t cloud-storage-frontend:latest .

docker run --rm -p 8080:80 \
  -e API_BASE_URL=https://gw.example.com \
  -e SHARE_BASE_URL=https://app.example.com \
  cloud-storage-frontend:latest
```

The image is multi-stage (Node→Nginx). Static config is baked into
`dist/` at build time, but `deployments/docker-entrypoint.sh` writes a
runtime `/config.js` from env vars at container start so the same image
can move between staging and prod without rebuilding.

For Kubernetes: `kubectl apply -f deployments/k8s/`. The frontend
ingress (`ingress.yaml`) pairs with `api-gateway/deployments/k8s/ingress.yaml`
on the same host so the SPA reaches the gateway via same-origin.

## Out of scope for this MVP

Per `frontend-blueprint.md` Appendix C:

- Trash page (deleted-only listing)
- Drag-n-drop reorder
- Server-side search (no endpoint)
- Profile page (no endpoint)
- Pinpoint cache invalidation
- Multi-tab BroadcastChannel sync
- Light theme

These are all hooks-in-place — adding them is mostly UI work without
backend changes.
